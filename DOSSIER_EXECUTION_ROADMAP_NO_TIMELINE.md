# Dossier Execution Roadmap (No Timeline, Invention Mode)

Reference docs (in precedence order):
1. `DOSSIER_FINAL_UNIFIED_ALGORITHM_PLAN.md` (authoritative algorithm spec)
2. `DOSSIER_PRODUCT_PLAN.md` (product context with Invention Mode override)

## Approach

Build by capability gates, not dates.
Optimize for **novel insight generation per token** with minimal coherence constraints.
Treat `BAEGC-lite` as runtime and `ADEPT-DS` as the invention kernel.
Keep the idea layer open and the structure layer fixed:
- wide generation at the hypothesis stage
- strict use of evidence packets, subgraph slices, assumptions, disconfirming signals, and score breakdowns at the packaging stage
- never reason over the raw full corpus once reduction and graphing have started

## Scope

- In:
  - evidence graph, synthesis triggers, hypothesis beam generation, mechanism synthesis, novelty scoring, counterfactual expansion, invention packaging
  - working context, compressed evidence memory, and reusable case memory
  - self-hosted BYOK defaults and budget controls
  - optional invention-style filters as controlled later-stage selectors
- Out:
  - strict trust-first filtering as primary objective
  - heavy RL stacks and broad multimodal/robotics expansion
  - Fogg-style behavior loops as the main control system
  - human-bias replication as the default reasoning mode
  - peak-end or memory-distortion modeling as a core requirement

### Current investigation flow (what `InvestigationService.investigate()` does today)

```
1. DefaultPlanner.create_plan(question)
   → returns InvestigationPlan with 2 hardcoded lanes (no LLM call)

2. For each lane:
   → search_provider.search(lane.query, lane, limit)
   → returns list[SearchHit] (title, url, snippet, content, metadata)

3. For each hit:
   → _build_packet() creates SourceAtom + EvidencePacket
   → repository.store_source() + repository.store_evidence_packet()

4. llm_client.synthesize(question, plan, packets)
   → DemoLlmClient returns 3 hardcoded RankedInsight objects
   → LiteLlmClient makes ONE completion() call, parses JSON for up to 3 insights

5. For each insight:
   → repository.store_insight()

6. Calculate InvestigationMetrics (tokens, cost, latency, grounded_novelty_ratio, coherence_failures)

7. render_dossier_markdown() → write_dossier() to data/dossiers/{case_id}.md

8. repository.complete_case()
```

### B3. Retrieval Reduction Pipeline (Chunking + Compression)

**Current state:** The full raw content from search hits is passed directly to the synthesis prompt. For long content, this wastes tokens and may exceed context windows.

**Target state:** A compression step between evidence collection and synthesis that: (a) chunks long content, (b) prunes irrelevant chunks, and (c) produces a compressed evidence summary that fits within a token budget.

**Detailed steps:**

[ ] B3.1. Add config fields to `Settings` in `src/dossier/config.py`:
  - `compression_enabled: bool` — default `True`.
  - `compression_max_tokens_per_packet: int` — default `500`. Maximum tokens for each evidence packet's content after compression.
  - `compression_max_total_tokens: int` — default `4000`. Maximum total tokens for all evidence going into synthesis.

[ ] B3.2. Create a new file `src/dossier/compression.py`:
  - Class `EvidenceCompressor`:
    - `__init__(self, settings: Settings)`.
    - `compress(self, packets: list[EvidencePacket]) -> list[EvidencePacket]`:
      1. For each packet, if `len(packet.summary) > compression_max_tokens_per_packet * 4` (rough char-to-token estimate): truncate the summary and quote to fit.
      2. Sort packets by `relevance` descending.
      3. Accumulate packets until `compression_max_total_tokens` is reached. Drop the rest (lowest relevance).
      4. Return the surviving (possibly truncated) packets.
      5. Log how many packets were dropped and how many chars were truncated.
  - Class `DemoCompressor`:
    - `compress()` returns the input packets unchanged (no-op for demo mode).
  - Factory function `build_compressor(settings: Settings)`:
    - If `settings.compression_enabled`, return `EvidenceCompressor(settings)`.
    - Otherwise, return `DemoCompressor()`.

[ ] B3.3. Integrate into `src/dossier/runtime.py`:
  - In `__init__`, create `self.compressor = build_compressor(self.settings)`.
  - In `investigate()`, after building all packets and before synthesis:
    - `compressed_packets = self.compressor.compress(packets)`

### B4. Parallel DAG Execution

**Current state:** Lanes are searched sequentially in a `for lane in plan.lanes:` loop inside `investigate()`.

**Target state:** Lanes run concurrently using Python's `concurrent.futures.ThreadPoolExecutor`. The rest of the pipeline remains sequential.

**Detailed steps:**

[ ] B4.2. In `src/dossier/runtime.py`, refactor the lane search loop in `investigate()`:
  - Replace the sequential `for lane in plan.lanes:` loop with:
    ```python
    from concurrent.futures import ThreadPoolExecutor, as_completed

    def _search_lane(self, case_id: str, lane: ResearchLane) -> list[EvidencePacket]:
        hits = self._safe_search(lane.query, lane)
        lane_packets = []
        for hit in hits:
            packet = self._build_packet(case_id=case_id, lane_id=lane.id, hit=hit)
            lane_packets.append(packet)
        return lane_packets
    ```
  - In `investigate()`:
    ```python
    packets: list[EvidencePacket] = []
    source_count = 0
    with ThreadPoolExecutor(max_workers=self.settings.max_parallel_lanes) as executor:
        futures = {
            executor.submit(self._search_lane, case_id, lane): lane
            for lane in plan.lanes
        }
        for future in as_completed(futures):
            lane = futures[future]
            try:
                lane_packets = future.result()
                for packet in lane_packets:
                    self.repository.store_evidence_packet(packet)
                packets.extend(lane_packets)
                source_count += len(lane_packets)
            except Exception:
                logger.exception("Lane %s failed", lane.name)
    ```
  - **Important:** The `SqliteRepository` must handle concurrent writes. Since SQLite uses WAL journal mode (already set), concurrent writes from threads in the same process are safe. But each thread should use the same connection via the shared `self.repository`. This works because Python's `sqlite3` module serializes access per connection.

## What You Need to Check Continuously

After every step, verify:

**Named failure modes to watch for (per Algorithm Plan):**
- **Empty novelty:** insights that score high on the formula but contain no real new information.
